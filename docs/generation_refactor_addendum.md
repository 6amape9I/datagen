# Addendum для Codex: правила поведения при рефакторинге generation layer

## 1. Не лечи старую схему — строй новый канонический слой
Не пытайся поддерживать два равноправных generation pipeline.
Нужен один канонический слой и два тонких entrypoint-а.

## 2. Экономия токенов — обязательный критерий
Каждое поле, попадающее в модель, должно быть оправдано.
Если поле не повышает качество выбора `syntactic_link_name`, не отправляй его.

## 3. Не смешивай single-sentence и multi-sentence задачи
Текущий generation layer размечает semantic relation для узлов внутри одного предложения.
Не добавляй сюда sentence-type metadata, эмоции и межпредложенческие связи.

## 4. Не тащи полную онтологию в каждый запрос
Нужен сжатый runtime context, а не энциклопедия.
Полные справочные документы могут лежать в репозитории, но не должны отправляться каждый раз модели целиком.

## 5. Ответ модели должен быть минимальным
Никогда не расширяй output schema сверх `id` + `syntactic_link_name`, если это не требуется новым решением владельца проекта.
Экономия токенов здесь важнее удобства “посмотреть всё сразу”.

## 6. Prompt builder должен собирать промпт из данных, а не из монолитной простыни
По возможности делай prompt composition модульным:
- core instructions
- compact ontology glossary
- task wrapper
- compact JSON payload

## 7. Не допускай расхождения между response schema и prompt ontology
Список разрешённых ролей должен быть единым источником истины.
Не дублируй enum вручную в нескольких местах.

## 8. Не хардкодь generation settings в provider-файлах
Thinking budget, max output tokens, mode/profile — из runtime config.
Provider должен быть transport adapter, а не второй config layer.

## 9. Удаляй мёртвый код
Если старый путь больше не нужен, не оставляй его “на всякий случай” в активном коде.
Максимум — thin wrapper с коротким redirect.

## 10. Любое усложнение должно быть оправдано
Если выбор между “чуть красивее” и “проще и дешевле”, выбирай проще и дешевле, если это не ухудшает качество разметки заметно.

## 11. Проверяй архитектуру по контрактам
При принятии решения спрашивай себя:
- помогает ли это выбрать `syntactic_link_name`?
- экономит ли это токены?
- сохраняет ли это простоту downstream?
- не смешивает ли это разные уровни анализа?

## 12. Не перепридумывай Stage 01
Считай compact `nodes[]` стабильным входным контрактом.
Generation layer должен адаптироваться к нему, а не тянуть Stage 01 обратно в рефакторинг.
